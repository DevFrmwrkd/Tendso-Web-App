## System Overview

### Core Concept

**Processing-First Withdrawals with Manual Wise Approval.** Creators withdraw instantly from their app balance. The Convex backend automatically creates the recipient, quote, and transfer in Wise (steps 1–3). The transfer then sits in the **Wise dashboard awaiting manual funding by the admin** (step 4). Once the admin funds it, Wise processes the payout and notifies the app via webhook, which marks the withdrawal as `completed`.

**Why manual approval?**
- Avoids Wise SCA (Strong Customer Authentication) key-signing complexity on the server.
- Gives the admin explicit control to review every payout before money leaves the Wise balance.
- Provides a natural fraud/mistake checkpoint.

## Wise Payment Flow (End-to-End)

### Complete Withdrawal Lifecycle

#### **Stage 1: Creator Initiates (Mobile App)**

Creator opens the Wallet tab and taps **Withdraw Funds**. If this is their first withdrawal, a **Wise Setup Modal** appears asking for their Wise email (saved via `withdrawals.saveWiseEmail` for future use). Subsequent withdrawals skip the setup modal and pre-fill the email.

Args sent to Convex:

```ts
await createWithdrawal({
  creatorId: creator._id,          // Id<"creators">
  amount: 300,                      // number (PHP)
  wiseEmail: "creator@email.com",  // string, validated client-side
});
```

**Client-side validation** ([ndm/app/(app)/(tabs)/wallet.tsx:141-213](../../app/(app)/(tabs)/wallet.tsx#L141-L213)):
- Email regex: `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`
- Amount: `> 0` and `<= creator.balance`
- Whitespace trimmed; email lowercased

**Response:**
```ts
{ withdrawalId: Id<"withdrawals"> }
```

No HTTP route is involved — the mobile app uses Convex's generated `useMutation(api.withdrawals.create)` hook directly. Auth is Clerk-based and attached automatically by the Convex client.

#### **Stage 2: Convex Processing (`withdrawals.create`)**

File: [ndm/convex/withdrawals.ts](../../convex/withdrawals.ts) lines 17–82. Exact sequence:

1. **Rate-limit check** — max **3 withdrawals per hour per creator**. Throws if exceeded.
2. **Validate amount** — must be `> 0`.
3. **Fetch creator** — 404 if not found.
4. **Balance check** — `creator.balance >= amount`, else throw `"Insufficient balance"`.
5. **Optimistic balance deduction** — `creator.balance -= amount` (locked against new withdrawals).
6. **Sanitize email** — `sanitizeEmail(wiseEmail)` strips stray characters.
7. **Insert withdrawal row:**
   ```ts
   {
     creatorId,
     amount,
     payoutMethod: "wise_email",
     accountDetails: sanitizedEmail,
     wiseEmail: sanitizedEmail,
     accountHolderName: creator.name,
     status: "pending",
     createdAt: Date.now(),
   }
   ```
8. **Schedule Wise transfer** — `ctx.scheduler.runAfter(0, internal.wise.initiateTransfer, { withdrawalId })`. Non-blocking; runs in background.
9. **Return** `{ withdrawalId }`.

**Not yet done at this stage:** no audit log, no push notification (these happen later on completion via webhook).

#### **Stage 3: Wise Transfer Creation (`wise.initiateTransfer` action)**

File: [ndm/convex/wise.ts](../../convex/wise.ts) lines 73–154. Runs asynchronously in the background.

1. **Read config** (lines 19–53):
   - `WISE_SANDBOX` toggle selects between sandbox and production env vars
   - Production: `WISE_API_TOKEN`, `WISE_PROFILE_ID`, `WISE_CREATOR_PAYOUT_BALANCE_ID`
   - Sandbox: `WISE_SANDBOX_TOKEN`, `WISE_SANDBOX_PROFILE_ID`, `WISE_SANDBOX_CREATOR_PAYOUT_BALANCE_ID`
2. **Create email recipient** — `services.wise.createEmailRecipient()` → `POST /v1/accounts` with `type: "email"`. Returns `recipient.id`.
3. **Create quote** — `services.wise.createQuote()` → `POST /v1/quotes` with source = target = PHP, amount, and optional `sourceBalanceId` (Creator Payout jar). Returns `quote.id`.
4. **Create transfer** — `services.wise.createTransfer()` → `POST /v1/transfers` linking `recipient.id` + `quote.id`. Reference = withdrawal ID. Returns `transfer.id`.
5. **Persist IDs** — `internal.withdrawals.setWiseTransferIds({ withdrawalId, wiseTransferId, wiseRecipientId, transactionRef })`. This moves status from `pending` → `processing`.
6. **Call `fundTransfer()`**:
   - **Production:** returns stub `{ type: "BALANCE", status: "WAITING_FOR_APPROVAL", errorCode: null }`. No Wise API call. Transfer waits in the Wise dashboard.
   - **Sandbox:** calls `GET /v1/simulation/transfers/{id}/processing`, which asks Wise to simulate processing so the webhook fires.

**Error handling** (lines 129–152):

| Failure point | Behavior |
|---|---|
| Steps 1–4 raise any error | `internal.withdrawals.markFailed` → balance restored, status = `failed` |
| Step 6 raises **403 SCA** | Swallowed as expected. Status stays `processing`. Admin will manually approve. |
| Step 6 raises any other error | `markFailed` + balance restored |

#### **Stage 4: Admin Manual Approval (Wise Dashboard)**

The transfer appears in [wise.com](https://wise.com) dashboard → **Payments / Transactions**. Admin:

1. Reviews the transfer (recipient email, amount, reference).
2. Clicks **Fund / Approve**.
3. **Selects the "Creator Payout" jar as the funding source** (Wise usually suggests this automatically because the quote was created with `sourceBalanceId = WISE_CREATOR_PAYOUT_BALANCE_ID`). Admin must **not** fund from the main PHP balance.
4. Wise debits the Creator Payout jar and pushes the email transfer.

### Creator Payout Jar

All creator withdrawals must be funded from the dedicated **Creator Payout** Wise jar, keeping creator payouts segregated from operating funds.

**Workflow:**
1. Admin tops up the Creator Payout jar from the main PHP balance as creators accumulate earnings.
2. When a creator withdraws, the backend sets `sourceBalanceId` on the quote (see [ndm/convex/wise.ts:96-103](../../convex/wise.ts#L96-L103)) to point to the jar.
3. Wise's dashboard suggests the jar when admin approves.
4. Admin confirms the source is **Creator Payout** before clicking Fund.

**Convex env vars (production):**
```bash
npx convex env set WISE_CREATOR_PAYOUT_BALANCE_ID "<jar_id>" --prod
```

**Sandbox:**
```bash
npx convex env set WISE_SANDBOX_CREATOR_PAYOUT_BALANCE_ID "<sandbox_jar_id>"
```

**Finding the jar ID:**
```bash
curl -H "Authorization: Bearer $WISE_API_TOKEN" \
  "https://api.wise.com/v4/profiles/$WISE_PROFILE_ID/balances?types=SAVINGS"
```
Look for the balance with `"name": "Creator Payout"` and copy its `id`.

If `WISE_CREATOR_PAYOUT_BALANCE_ID` is not set, the transfer falls back to the main balance. This fallback is acceptable in dev but **must not be used in production**.

#### **Stage 5: Webhook Updates Status**

Wise sends webhook to `https://<convex-deployment>.convex.site/wise-webhook`.

File: [ndm/convex/http.ts](../../convex/http.ts) lines 115–281.

1. **Rate limit** — max **60 events per minute** (via `checkWebhookRateLimit`). Returns 429 if exceeded.
2. **Signature verification** — `X-Signature-SHA256` header, RSA-SHA256 against `WISE_WEBHOOK_PUBLIC_KEY`. If the env var is not set, verification is skipped with a warning log (dev-only convenience).
3. **Event dispatch** by `event_type`:

   | Event type | Handler | Purpose |
   |---|---|---|
   | `transfers#state-change` | Outgoing transfer handler (lines 206–265) | Creator withdrawal state updates |
   | `balances#credit` | Incoming payment handler (lines 184–204) | Business-owner deposit auto-credit |

4. **State mapping** for `transfers#state-change`:

   ```ts
   {
     processing:           "processing",
     outgoing_payment_sent: "completed",
     cancelled:            "failed",
     funds_refunded:       "failed",
     bounced_back:         "failed",
     charged_back:         "failed",
   }
   ```

5. **Update via transaction ref** — `internal.withdrawals.updateByTransactionRef` looks up the withdrawal by `transactionRef` (Wise transfer ID) using the `by_transactionRef` index, then applies the new status.

   - **On `completed`:** increments `creator.totalWithdrawn` by the withdrawal amount; sends "Withdrawal Completed!" push notification.
   - **On `failed`:** restores `creator.balance` by the withdrawal amount; logs reason in `failureReason`.

#### **Stage 6: Creator Receives Funds**

Timeline:

| Moment | State in app | State in Wise |
|---|---|---|
| Creator taps Confirm | balance deducted; row = `pending` | — |
| `initiateTransfer` completes steps 1–4 | row = `processing`; IDs stored | transfer exists, awaiting funding |
| Admin clicks Fund in Wise dashboard | unchanged | transfer enters outgoing queue |
| Wise sends money to creator's Wise email | unchanged | `outgoing_payment_sent` |
| Wise webhook fires | row = `completed`; `totalWithdrawn` incremented; push sent | `outgoing_payment_sent` |

Mobile app auto-updates via Convex reactive query on `creators.balance` and `withdrawals.getByCreator`.

#### **Stage 6b: Wise Email Payment Mechanics — How the Creator Claims Their Payout**

Tendso uses Wise's **send-to-email** product, not a direct bank transfer. The recipient on the Wise side is identified only by their email address — we never collect or store bank account numbers. This has real implications for the creator's experience, especially on their first payout.

##### How Wise routes money sent to an email

When [`wise.initiateTransfer`](../../convex/wise.ts) calls `POST /v1/accounts` with `type: "email"`, Wise stores the recipient as an "email recipient" — not yet tied to a real bank account. When the admin funds the transfer from the Creator Payout jar, Wise branches based on whether the email already has a Wise account.

###### Case A — Creator already has a Wise account with the same email

- Wise credits the money **directly to their Wise PHP balance within minutes**.
- The creator receives an email from Wise: *"You received ₱X from Tendso"*.
- **No invitation link step.** The money is already in their Wise account.
- Our webhook fires with `outgoing_payment_sent`; our row flips to `completed`.

###### Case B — Creator does NOT have a Wise account (or used a different email)

- Wise sends an **invitation email** to the recipient address.
- Sender: `noreply@wise.com` (or `no-reply@transferwise.com` in some regions).
- Subject line varies by locale, commonly: *"Tendso sent you money"* or *"You've received money from Tendso"*.
- Body contains a **claim link that Wise generates per transfer**. We don't construct this link — Wise owns the URL, rotates the token per transfer, and manages its expiry. Typical format observed is under the `wise.com` domain (e.g., `https://wise.com/claim/<unique-token>`), but the exact URL should not be hardcoded or anticipated on our side. **The only authoritative source for the link is the email Wise sends to the recipient.**
- The creator clicks the link, then either signs up for a free Wise account or provides their bank details so Wise can deposit the money directly — Wise supports both paths.
- The **claim window is 7 calendar days** (per Wise's current help docs — see [How can I send money to an email address?](https://wise.com/help/articles/2932105/how-can-i-send-money-to-an-email-address)). If the creator does not claim within 7 days, Wise **automatically cancels and refunds** the transfer to our Creator Payout jar. The webhook fires with state `funds_refunded`; our code marks the withdrawal `failed` and restores the balance to the creator's in-app wallet.

##### Creator-facing step-by-step (first-time payout)

1. **In Tendso app:** creator taps **Withdraw Funds** → enters ₱300 → enters their Wise email → confirms. Balance deducts immediately. Row appears in history as `pending` → `processing`.
2. **Within minutes** (after our admin approves in Wise): creator receives an email from `noreply@wise.com` (or `no-reply@transferwise.com`) titled *"Tendso sent you money"*.
3. **Creator opens the email and clicks "Claim your money":**
   - **Has a Wise account:** logs in → money appears in their PHP balance within minutes.
   - **Doesn't have Wise:** signs up (free) → completes ID verification (photo of ID + selfie) → money appears once verified (usually minutes, up to 2 business days for manual review).
4. **Creator can then:**
   - Keep it in their Wise PHP balance.
   - Convert to USD or other currencies (Wise handles FX).
   - Send to their Philippine bank account (BDO, BPI, UnionBank, Metrobank, etc.).
   - Cash out to **GCash** — Wise supports direct GCash transfers in the Philippines.

##### Important caveats the creator should know

- **The email they enter in our app must match their Wise account email**, OR they must be willing to sign up with that email. Typos like `example@gmal.com` instead of `example@gmail.com` cause the invitation to go nowhere and the transfer auto-refunds after 14 days.
- **New Wise users must complete KYC before funds release.** Acceptable Filipino IDs per BSP / Wise policy: PhilID, driver's license, passport, UMID, SSS, or PRC license. They'll also need a selfie photo.
- **A Wise account is free.** No signup fee, no monthly fee. Wise makes its revenue on currency conversion, which doesn't apply here since we pay PHP → PHP.
- **Expect a short delay the first time.** Even Case A isn't instant — Wise moves funds within its own rails and the creator may see anywhere from 1 minute to ~30 minutes between our admin clicking Fund and the money landing.

##### If the invitation is ignored or the email is wrong

Timeline for an unclaimed invitation (7-day window, per Wise's current policy):

| Day | Event | In-app withdrawal status |
|---|---|---|
| 0 | Admin funds the transfer; Wise sends invitation email | `processing` |
| 1–6 | Invitation sits in the creator's inbox unclaimed | `processing` (still) |
| 7 | Wise cancels the transfer, refunds to our Creator Payout jar | — (Wise-side event) |
| 7 (moments later) | Webhook fires with `funds_refunded` | `failed` |
| 7 (same moment) | Our webhook handler restores `creator.balance` | — |

The creator sees their balance back in the app and can initiate a new withdrawal with the correct email.

> **Why we can't show the claim link in our app:** The claim link is embedded in Wise's outgoing email and is tied to Wise's security/auth model — we have no API to fetch it. We see only the `transfer.id`, `state`, and webhook events. If a creator asks "where's my link?", direct them to search their inbox for `noreply@wise.com` or `no-reply@transferwise.com`.

**Admin-side indicator:** transfers stuck at `processing` in the Wise dashboard past their expected settlement time often mean the recipient hasn't claimed. Admins can inspect the transfer in Wise → *Details* → *Tracking* to see if the invitation was delivered, opened, or clicked.

##### Why email-based payment (not direct bank transfer)

| Concern | Email transfer (current) | Direct bank transfer (alternative) |
|---|---|---|
| Data we collect / store | Email only | Bank name, account number, branch, routing — PCI / Philippine DPA scope |
| Recipient choice of destination | Creator picks — Wise balance, bank, or GCash | We pick — money lands in one account we specified |
| Setup friction for us | None — just send to email | We must validate bank routing per recipient |
| First-time creator friction | 14-day window to sign up for Wise + KYC | Must give us bank details up front |
| Cross-border support | Built in — Wise handles FX | Country-specific bank integrations |
| Recipient identity verification | Wise does it (KYC) | Our responsibility to verify |

For Tendso's current scope — Filipino creators getting paid in PHP — email transfer is the right tradeoff. Most Filipino freelancers already have Wise, Payoneer, or similar accounts, and for those who don't, the one-time Wise signup gives them a useful ongoing tool (multi-currency account, free PH bank transfers, GCash cash-out).

### `updateStatus` behavior

| Target status | Side effects |
|---|---|
| `completed` | `totalWithdrawn += amount`, push "Withdrawal Completed!", audit log |
| `failed` | `balance += amount` (restore), push failure notice, audit log with reason |
| `processing` | Status update only, audit log |

`updateStatus` is the escape hatch when the Wise webhook doesn't fire (misconfigured subscription, environment mismatch, etc.) or when admin needs to mark a stuck transfer as complete after confirming it went through. **There is no separate `retry` mutation** — retry = new withdrawal after fixing the root cause, or a manual `updateStatus` override.

## Error Handling & Retry Logic

### Failure scenarios (current code)

| Failure | Where detected | Behavior |
|---|---|---|
| Rate limit (>3/hr) | `withdrawals.create` step 1 | Throws; mobile shows error toast. Nothing persisted. |
| Invalid amount (`<= 0`) | `withdrawals.create` step 2 | Throws. |
| Insufficient balance | `withdrawals.create` step 4 | Throws. |
| Wise API rejects recipient/quote/transfer | `wise.initiateTransfer` | `markFailed` → balance restored, status = `failed` |
| Wise SCA 403 on `fundTransfer` | `wise.initiateTransfer` | Swallowed. Status stays `processing` — admin approves manually. |
| Wise returns `bounced_back` / `cancelled` / etc. via webhook | `updateByTransactionRef` | Balance restored, status = `failed`, `failureReason` stored. |
| Webhook never fires | — | Row stuck at `processing`. Admin uses `updateStatus` to mark completed after verifying in Wise dashboard. |
| Webhook signature invalid | `http.ts` | Returns 401; no state change. |

### Automatic retry

There is **no automatic retry** in the current implementation. The code philosophy is:

- Steps 1–4 (recipient / quote / transfer) are **idempotent per withdrawal** in Wise's model (each withdrawal produces a unique `customerTransactionId`), but the app does not loop on failure — a Wise error marks the withdrawal failed and restores balance. The creator can simply initiate a new withdrawal.
- Step 6 (funding) is intentionally manual in production. The admin is the "retry" operator.

### Manual recovery

| Symptom | Recovery action |
|---|---|
| Row stuck at `processing` for >24h | Check Wise dashboard; if admin funded: `updateStatus → completed`. If not: click Fund. |
| Row stuck at `pending` for >1 min | `initiateTransfer` never ran. Check Convex logs + reschedule manually if needed. |
| Webhook misconfigured | Set webhook URL in Wise dashboard + `WISE_WEBHOOK_PUBLIC_KEY` env var + re-subscribe to "Transfer update events". |
| Creator's Wise email wrong | After transfer fails, creator updates email + submits new withdrawal. |

## Security & Compliance

### Data handling

**At rest (Convex):**
- Wise emails stored in cleartext on `creators.wiseEmail` and `withdrawals.wiseEmail` (needed to contact the Wise payee and surface in history).
- Transfer IDs stored in `wiseTransferId` / `wiseRecipientId` / `transactionRef` for webhook mapping and support.
- Balances and `totalWithdrawn` stored numerically — no rounding cash equivalents; every amount is integer-PHP.

**In transit:**
- Mobile → Convex: TLS via Convex client (HTTPS).
- Convex → Wise: HTTPS, Bearer token in Authorization header.
- Wise → Convex webhook: HTTPS + RSA-SHA256 signature in `X-Signature-SHA256`.

**Secrets:**
- `WISE_API_TOKEN`, `WISE_PROFILE_ID`, `WISE_WEBHOOK_PUBLIC_KEY`, `WISE_CREATOR_PAYOUT_BALANCE_ID` live only in Convex env (`npx convex env set ... --prod`). Never committed.
- Sandbox analogs use separate vars so prod and sandbox can coexist on the same deployment if needed (`WISE_SANDBOX_*`).

### Authentication & authorization

**Mobile:**
- Clerk authenticates every Convex request.
- `withdrawals.getByCreator` checks that the caller's Clerk identity matches `creatorId`.
- `withdrawals.create` performs the same ownership check via the creator record.

**Admin:**
- `withdrawals.getAll`, `getByStatus`, `updateStatus` all require an admin role check (see helpers in [ndm/convex/lib/auth.ts](../../convex/lib/auth.ts)).
- All admin overrides write to `auditLogs` with admin ID, timestamp, and reason.

### Rate limits (server-side)

| Scope | Limit | Source |
|---|---|---|
| Withdrawal creation per creator | **3 / hour** | [ndm/convex/lib/rateLimit.ts](../../convex/lib/rateLimit.ts) via `withdrawals.create` |
| Wise webhook events | **60 / minute** | [ndm/convex/lib/webhookRateLimit.ts](../../convex/lib/webhookRateLimit.ts) |

### Audit logging

**On `updateStatus` (admin path):** logs `{ action: "payment_sent", adminId, withdrawalId, amount, payoutMethod, transactionRef, status, failureReason? }`.

**On `create` and webhook path:** no audit log rows (intentional — the withdrawal row itself is the record of truth; audit logs are reserved for admin intervention).

### Compliance posture

- **PCI DSS:** not applicable — no card data stored or processed. Wise is PCI-compliant and handles any card-adjacent flows on their side.
- **KYC/AML:** Wise performs KYC on both the sender (admin's Wise business account) and recipient. The app does not independently verify creator identity for withdrawal purposes beyond the Clerk login.
- **Data retention:** withdrawal rows are retained indefinitely for tax / dispute purposes. Soft-delete of a creator account retains withdrawal history for compliance.
- **GDPR / Philippine DPA:** creators can request data export via `withdrawals.getByCreator` output + profile export (not yet automated; handled on request).

## Troubleshooting

| Symptom | Most likely cause | Fix |
|---|---|---|
| Withdrawal stuck at `processing` >24h | Admin hasn't funded in Wise dashboard | Open Wise → Payments → find by `wiseTransferId` → Fund from Creator Payout jar |
| Withdrawal stuck at `pending` | `wise.initiateTransfer` action failed to run | Check Convex logs; re-run via admin or file a manual override |
| "Insufficient balance" on a valid amount | Creator's balance out of sync after failed withdrawal | Check `creators.balance`; previous failed flow may not have restored correctly — verify in audit logs |
| Wise API rejects quote | Invalid `sourceBalanceId` | Confirm `WISE_CREATOR_PAYOUT_BALANCE_ID` matches a real jar on the configured profile |
| Webhook 401 | `WISE_WEBHOOK_PUBLIC_KEY` mismatch | Re-copy the public key from Wise settings |
| Webhook 429 | 60/min burst exceeded | Investigate — Wise should not burst this high under normal operation |
| Creator "balance didn't update after completion" | Mobile query cache stale | Pull to refresh the Wallet tab |
